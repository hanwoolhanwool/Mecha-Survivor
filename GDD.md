# Mecha Survivor — 게임 기획서 (GDD v1)

- 작성일: 2026-07-13
- 대상 빌드: **v1 (수직 슬라이스 → 플레이 가능한 1개 런)**
- 기술 환경은 `SETUP.md` 참조. 이 문서는 **"이 게임이 무엇인가"** 만 정의한다.

## 1. 한 줄 정의

> **20분간 하늘을 날며, 쿨다운이 돌아오는 무기를 번갈아 조준 사격하고,
> 레벨업마다 메카를 조립·개조해 점점 거세지는 적 무리에서 살아남는 3D 슈터.**

### 1.1 다른 뱀서라이크와 무엇이 다른가 (핵심 축)

| 축 | 정통 뱀서라이크 | **Mecha Survivor** |
|---|---|---|
| 공격 | 자동 발사 | **마우스 수동 조준 사격** |
| 전투 중 플레이어의 일 | 이동(위치 잡기)뿐 | **이동 + 조준 + 무기 로테이션** |
| 공간 | 2D 평면 | **3D 공중전 (Y축 활용)** |
| 재미의 위치 | 빌드 설계 (전투 전) | **빌드 설계 + 전투 중 운용** |

**결론:** 뱀서라이크에서 **성장 구조만** 빌려오고, 전투는 액션 슈터다.
따라서 설계 목표는 **"손이 놀지 않게 하는 것"** 이며, 그 유일한 장치가 **쿨다운 로테이션**이다.

### 1.2 핵심 재미 루프

```
적 무리를 향해 비행 → 사거리/각도 잡기 → 무기 A 발사
   → A 쿨다운 진입 → 그동안 무기 B·C 로 전환해 계속 압박
      → A 쿨 복귀 → 다시 A
         → 적 처치 → 경험치 → 레벨업 → 3택 → 메카가 강해짐
            → 더 많은 적 → 반복
```

**이 루프에서 "무기 B로 전환"하는 순간이 게임의 정체성이다.**
한 무기만 계속 난사할 수 있으면 게임은 무너진다.
→ 3장·4장의 모든 규칙은 **이 명제를 지키기 위해** 존재한다.

## 2. 플레이어 — 이동과 조작

### 2.1 조작 매핑

| 입력 | 동작 |
|---|---|
| `W A S D` | 수평 이동 (카메라 기준) |
| `Space` | 상승 (홀드) |
| `Shift` | 하강 (홀드) |
| 마우스 이동 | 카메라 회전 + 조준 |
| 좌클릭 | 무기 슬롯 1 발사 |
| 우클릭 | 무기 슬롯 2 발사 |
| `Q` / `E` | 무기 슬롯 3 / 4 발사 (파츠로 해금 시) |
| `R` | (예약) 필살기 / 오버드라이브 |
| `V` | **카메라 시점 전환 (1인칭 ↔ 3인칭)** |
| `Esc` | 일시정지 |

### 2.2 이동 모델 — "공중이 기본, 지상은 선택"

- **기본 상태는 비행(Flight).** 스폰 시 공중에 떠 있으며, 비행에 자원 소모는 없다.
- **고도 상한(Ceiling) 존재.** 무한 상승 불가.
  - *이유:* 위로 무한정 도망칠 수 있으면 모든 근접 적이 무의미해진다.
    천장은 **"도망칠 곳이 없다"** 를 보장하는 장치다.
- **지상 착지 및 보행 가능.** 하강해 바닥에 닿으면 보행 상태(Grounded)로 전환.

#### 지상 상태의 존재 이유 (설계 핵심)

비행이 공짜인 이상, 플레이어가 내려올 이유를 **명시적으로 만들지 않으면 지상 기능은 죽는다.**
이 게임의 유일한 자원 축이 **쿨다운**이므로, 보상도 쿨다운으로 준다.

> ### **착지·보행 중에는 모든 무기의 쿨다운이 더 빨리 돈다 (× 1.5 ~ 2.0)**

- **공중** = 안전(회피 자유) 하지만 **화력 회전이 느림**
- **지상** = 화력 회전이 빠르지만 **위험(지상 잡병에 포위, 회피 제한)**

이로써 Y축은 단순 이동이 아니라 **"안전 ↔ 화력"의 저울**이 된다.
새 자원을 추가하지 않고 기존 쿨다운 축만으로 지상의 가치를 만든다는 점이 중요하다.

### 2.3 로봇적 움직임 — "표현은 기계처럼, 반응은 즉각적으로" ★

이 게임은 **핵앤슬래시**다. 수백 마리 사이를 헤집는 게임에서 **입력이 굼뜨면 그냥 나쁜 게임**이다.

> ### 절대 원칙
> **관성·가속 램프·조작 지연을 넣지 않는다.**
> `WASD`를 누르면 **즉시** 최고속으로 움직이고, 떼면 **즉시** 멈춘다.
> 마우스를 돌리면 **즉시** 그곳을 조준한다.
>
> **로봇다움은 입력 응답을 늦춰서 만드는 것이 아니다.**
> 그건 "무거운 기계"가 아니라 **"조작이 안 먹는 게임"** 이다.

#### 그러면 로봇다움은 어디서 나오는가 — 표현 레이어

**시뮬레이션(속도·위치·조준)은 즉각적이고, 메시·VFX·사운드가 그것을 기계처럼 해석한다.**
아래 요소들은 **전부 게임플레이에 0의 지연도 주지 않는다.** 눈과 귀에만 작용한다.

| 요소 | 내용 | 지연 발생? |
|---|---|:---:|
| **상하체 분리** ★ | 하체는 **이동 방향**, 상체(포탑)는 **조준 방향**. 옆으로 날며 뒤를 쏜다 | **없음** (아래 참조) |
| **관절 단위 회전** | 몸통이 통째로 홱 돌지 않는다. 허리 → 어깨 순으로 **드르륵** 따라온다 | 없음 (시각만) |
| **스러스터 분사** | 이동·상승 시 **해당 방향의 노즐**이 실제로 불을 뿜는다. 움직임에 물리적 원인이 보인다 | 없음 |
| **호버링 아이들** | 정지 중에도 미세하게 상하 부유 + 자세 보정 스러스터가 툭툭 튄다. **떠 있는 기계** | 없음 |
| **기계음** | 서보 모터, 유압, 금속 마찰. **로봇다움의 절반은 소리다** | 없음 |
| **착지 반응** | 접지 순간 무릎 굽힘 + 먼지 + 짧은 카메라 킥 (**입력은 계속 받는다**) | 없음 |
| **보행 IK** | 지상 보행 시 다리가 지형에 맞게 딛는다. 공중에선 랜딩기어처럼 접힌다 | 없음 |

#### 상하체 분리 — 지연 없이 구현하는 법

메카의 정체성이지만, 잘못 만들면 **조작 지연의 주범**이 된다.

- **총구는 즉시 조준선을 향한다.** 발사 판정은 **언제나 조준선(크로스헤어) 기준**이다.
  → 상체 회전이 덜 끝났다고 발사를 막지 **않는다.** 막는 순간 답답한 게임이 된다.
- **상체 메시의 회전은 순수 시각 연출**이다. 조준선을 **빠르게**(그러나 관절답게 단계적으로) 따라간다.
  → 플레이어는 "포탑이 드르륵 돌아간다"를 보지만, **총알은 이미 크로스헤어로 나갔다.**

> **이 분리가 이 게임 로봇 표현의 핵심 트릭이다.**
> 보이는 것(기계적, 관절이 따라옴)과 실제 판정(즉각적)을 **분리**해서 둘 다 얻는다.

#### 코드 구조로 강제한다

```
MechaController  — 입력 → 속도/위치/조준. 지연 0. 관성 0.
      │ (상태를 읽기만 함)
      ▼
MechaVisuals     — 메시 회전, 스러스터 VFX, IK, 사운드, 애니메이션
                   여기서는 얼마든지 보간·지연해도 된다 (게임플레이와 무관)
```

**게임플레이 코드는 `MechaVisuals`의 존재를 모른다.** 시각이 아무리 늦게 따라와도 판정은 이미 끝나 있다.
→ 연출을 마음껏 화려하게 만들 수 있고, **연출 튜닝이 조작감을 절대 망치지 못한다.**

### 2.4 카메라 — 1인칭 / 3인칭 이중 모드

**두 시점을 모두 구현하고, 인게임에서 즉시 전환할 수 있게 한다** (기본 `V` 키).
개발 중에는 **비교 테스트 수단**이고, 출시 시에는 **플레이어 옵션**이 된다.

#### 절대 원칙 — 조준은 시점에 종속되지 않는다

> **어떤 시점이든 조준은 "화면 중앙 크로스헤어 → 카메라 정면 레이캐스트"로 동일하다.**

시점에 따라 명중률이 달라지면 두 모드는 "시점 선택"이 아니라 **서로 다른 게임**이 되고,
밸런싱이 두 벌 필요해진다. 따라서 조준 담당(`MechaAimer`)은 **카메라가 1인칭인지 3인칭인지 모른다.**
카메라에게 "지금 네 시선 원점과 방향이 뭐냐"만 묻는다.

- 대량의 적을 상대하므로 **완전한 정밀 조준을 요구하지 않는다.**
  무기별로 관용도를 둔다 (탄 퍼짐, 약한 유도, 폭발 범위 — 3.3 참조).

#### 1인칭 (First Person) — "정확함"

- 콕핏 시점. 메카 본체는 보이지 않거나 일부(팔/무기)만 보인다.
- **장점:** 조준이 직관적, 몰입감, 속도감이 강함
- **단점:** 3D 공중전에서 **주변 상황 파악이 어렵다** — 사방에서 오는 적을 놓친다
- 역할: **정밀 조준 테스트용 기준선.** "조준이 이상하다"는 느낌이 들 때, 1인칭에서도 이상하면 조준 로직 문제, 1인칭에선 멀쩡하면 3인칭 카메라 문제다. **디버깅 도구로서의 가치가 크다.**

#### 3인칭 (Third Person) — "역동적" ★ 주력

메카의 어깨 뒤. **이 게임의 기본 시점이며, 여기에 연출 예산을 쏟는다.**

역동성은 "흔들면 된다"가 아니다. **속도와 기동을 눈으로 읽히게 만드는 것**이다.
아래 요소들은 전부 **플레이어의 현재 속도·입력에 반응**한다.

| 요소 | 동작 | 목적 |
|---|---|---|
| **스프링 암 지연 (Camera Lag)** | 가속하면 카메라가 살짝 **뒤처졌다가** 따라붙는다 | 속도감. **기체는 이미 즉시 움직였고, 카메라만 따라온다** — 조작은 전혀 느려지지 않는다 |
| **속도 기반 FOV** | 저속 60° → 고속 75° 로 **부드럽게** 변화 | 빠를수록 시야가 넓어지고 화면 가장자리가 흘러 속도감 발생 |
| **뱅킹 / 롤 (Banking)** | 좌우 이동 시 카메라를 진행 방향으로 **살짝 기울인다** (±10° 내외) | 비행체 특유의 선회감. **3D 비행에서 가장 효과가 큰 연출** |
| **상승/하강 피치 오프셋** | 상승 시 카메라가 약간 아래에서 올려다보고, 하강 시 위에서 내려다본다 | Y축 이동을 시각적으로 체감 |
| **충돌 회피 (Camera Collision)** | 카메라와 메카 사이에 지형이 끼면 카메라를 **당긴다** | 착지 시 지면에 카메라가 파묻히는 것 방지 (**필수**) |
| **셰이크** | 피격 / 대형 무기 발사 시 짧은 흔들림 | 타격감 |

> **주의 — 역동성의 상한:** 위 효과들이 **조준을 방해하면 즉시 실패**다.
> 이 게임은 수동 조준 슈터다. 크로스헤어가 흔들려 못 맞히면 연출이 아니라 버그다.
> 따라서 **모든 효과는 개별로 강도 조절(0~1)이 가능해야 하고, 전부 0으로 끄면 정적인 카메라가 되어야 한다.**
> → 강도값은 인스펙터에 노출해 **플레이하며 조정**한다.

#### 구현 방침

**Cinemachine을 쓰지 않고 직접 구현한다.**
- 위 효과(뱅킹, FOV 펀치, 속도 연동)는 결국 Cinemachine Extension을 따로 짜야 해서 **이득이 적다.**
- 카메라가 게임의 핵심 손맛에 직결되므로 **전체 제어권을 갖는 편이 낫다.**
- 의존성이 줄어 학습·디버깅이 쉽다.

## 3. 전투 — 쿨다운 로테이션

이 게임의 **가장 중요한 시스템**이다. 무기에 **탄약도, 에너지 소모도 없다. 오직 쿨다운뿐이다.**

### 3.1 규칙

- 플레이어는 **무기 슬롯을 여러 개** 가진다 (초기 2개 → 파츠로 최대 4개까지 확장).
- 각 무기는 **독립적인 쿨다운**을 가진다.
- 쿨다운이 돌아온 무기는 **언제든, 몇 번이든 쏠 수 있다.** (자원 검사 없음)
- **핵심 규칙:** 어떤 무기도 **혼자서는 지속 화력을 낼 수 없다.**
  → 무기의 (사용 시간 : 쿨다운) 비율을 반드시 **1보다 훨씬 작게** 잡는다.
  → 그 결과 생기는 빈 시간을 다른 무기로 메우는 것이 **로테이션**이며, 이것이 전투의 리듬이다.

> **왜 자원을 두지 않는가?**
> 자원(에너지/탄약)을 얹으면 "쏠지 말지"를 고민하게 되어 **손이 멈춘다.**
> 이 게임이 원하는 건 **쉼 없이 조준하고 쏘되, 쏘는 대상(무기)이 계속 바뀌는 것**이다.
> 쿨다운만 두면 플레이어의 질문은 "쏠까 말까"가 아니라 **"지금 어느 무기를 쏠까"** 가 된다.

### 3.2 쿨다운 감소(쿨감) — 나눗셈 모델 ★

쿨감은 성장의 3대 축 중 하나다(4.2 "에너지 자원" 카테고리). **계산식이 게임의 생사를 가른다.**

```
실제 쿨다운 = 기본 쿨다운 / (1 + 쿨감수치)
```

| 쿨감 누적 | 실제 쿨다운 (기본 6초) | 발사 횟수 |
|:---:|:---:|:---:|
| 0% | 6.00초 | ×1.0 |
| +100% | 3.00초 | ×2.0 |
| +200% | 2.00초 | ×3.0 |
| +400% | 1.20초 | ×5.0 |

**절대 채택하면 안 되는 방식:**

| 방식 | 식 | 왜 안 되는가 |
|---|---|---|
| 뺄셈 | `쿨다운 - 감소량` | 쿨감을 쌓으면 **쿨다운이 0 또는 음수**가 된다 |
| 퍼센트 합산 | `쿨다운 × (1 - 합산%)` | 합산이 100%에 닿는 순간 **쿨다운 0** → 무한 난사 → **로테이션 붕괴 = 게임 붕괴** |

나눗셈 모델은 쿨다운이 **0에 수렴하되 절대 도달하지 않는다.**
덕분에 "쿨감 상한 70%" 같은 인위적 캡을 관리할 필요가 없고, 플레이어 체감도 정직하다
(쿨감 100%를 더 쌓으면 발사 횟수가 정확히 1배씩 늘어난다).

**적용 순서:**
```
실제 쿨다운 = 무기 기본 쿨다운
              / (1 + 전역 쿨감 + 해당 무기 전용 쿨감)
              / (지상 착지 중이면 1.5 ~ 2.0, 아니면 1.0)
```

### 3.3 무기 역할 분류 (서로를 메우도록)

| 역할 | 쿨다운 | 특징 | 로테이션에서의 자리 |
|---|---|---|---|
| **주력 (Sustain)** | 짧음 | 낮은 데미지, 연사 | 로테이션의 **바닥**. 빈 시간을 메운다 |
| **화력 (Burst)** | 중간 | 무리 정리, 범위 | 무리가 뭉쳤을 때 **터뜨린다** |
| **결정타 (Finisher)** | 김 | 단일 대상 고데미지 | 엘리트·보스 **삭제** |
| **유틸 (Control)** | 김 | 위치 선점, 군중 제어 | 상황을 **만든다** |

> **역할이 겹치면 로테이션이 무너진다.** 무기를 늘릴 때는 **역할이 비어 있는지** 먼저 확인한다.

### 3.4 무기 목록 — 화려함이 포인트 ★

> ### 설계 표어
> **"강해지는 것이 눈에 보여야 한다."**
> 이 게임의 쾌감은 데미지 숫자가 아니라 **화면이 점점 난장판이 되는 것**이다.
> 업그레이드는 수치만 올리지 않는다 — **연출 자체를 키운다.**

### 🟢 주력 (Sustain) — 로테이션의 바닥

#### 1. **개틀링 / 발칸** (v1 필수)
- 6연장 총열이 **예열되며 회전**하다가 발사. 멈출 때도 관성으로 **천천히 멈춘다**
- 총구 화염 + **탄피가 우수수 쏟아져** 아래로 떨어짐 (물리 X, 순수 시각)
- **트레이서**: 5발 중 1발만 빛나는 예광탄 → 탄 흐름이 선으로 보인다
- **강화 시:** 총열 수 증가 → 연사 속도 → **Lv.5: 쏠수록 빨라지는 가속 회전 (개틀링의 로망)**

#### 2. **레이저 캐논** (연발형)
- 짧은 펄스 레이저를 **따다다닥** 연사. 즉시 명중(히트스캔)이라 조준이 편함
- 맞은 자리에 **작은 스파크 + 그을음**
- **강화 시:** 동시 발사 라인 2→3→4갈래로 **부채꼴 확산**

---

### 🟠 화력 (Burst) — 무리를 지운다

#### 3. **미사일 포드** ★★ (v1 필수 — 이 게임의 간판 연출)

> **"미사일 여러 발이 궤적을 그리며 나간다"** — 요청하신 핵심 연출.
> 그냥 여러 발 쏘는 게 아니라 **4단 연출**로 만든다.

```
① 발사   — 어깨/등 해치가 열리고 미사일이 [수직으로 팝업]
② 정렬   — 공중에서 잠깐 멈칫하며 노즈를 목표로 돌린다  ← 여기가 화려함의 8할
③ 유도   — 각 미사일이 [서로 다른 곡선]으로 흩어졌다가 목표로 수렴
            → 하얀 연기 트레일이 하늘에 [나선/S자 궤적]을 그린다
④ 착탄   — 동시가 아니라 [0.05초씩 시차]를 두고 연쇄 폭발 (두두두두둥)
```

- **핵심 트릭 ②·③:** 미사일이 곧게 날아가면 하나도 안 화려하다.
  **일단 위로 튀어오른 뒤 곡선으로 꺾여 들어가야** 궤적이 그려진다.
- **핵심 트릭 ④:** 동시 착탄은 "펑" 한 번이지만, 시차 착탄은 **"두두두두둥"** 이 된다. 같은 데미지, 3배의 쾌감.
- **강화 시:** 4발 → 8발 → 16발 → **Lv.5: 화면이 연기 궤적으로 뒤덮인다**

#### 4. **산탄 캐논 (스프레드)**
- 근거리 광역. **강력한 반동으로 기체가 뒤로 밀린다** (공중에서 후퇴기로 활용 가능)
- 발사 시 **원뿔형 충격파 링**이 퍼지고, 맞은 적들이 **뒤로 날아간다** (넉백)
- **강화 시:** 탄 수 증가 → **Lv.5: 발사 시 화염 방사 형태로 변화**

#### 5. **클러스터 폭탄**
- 하나를 던지면 공중에서 **터지면서 새끼 폭탄 다수로 분열** → 넓은 지역 융단폭격
- **강화 시:** 분열 횟수 증가 → **Lv.5: 2차 분열 (새끼가 또 터진다)**

---

### 🔴 결정타 (Finisher) — 화면을 지배한다

#### 6. **대출력 빔 (Beam Cannon)** ★★ (v1 필수)

> **"빔의 크기가 크다"** — 요청하신 연출. **차징이 화려함의 근원**이다.

```
① 차징   — 가슴/등의 코어가 [빛나며 링이 수축]. 주변 파편이 빨려든다. 저음 웅- 소리
           (약 1초 — 이 예고 동작이 "무언가 온다"를 만든다)
② 발사   — [굵은 관통 빔]. 이중 구조:
              · 코어 = 새하얀 얇은 심지
              · 외피 = 반투명 넓은 글로우 (여기가 "크다"를 만든다)
③ 지속   — 빔이 조준을 따라 [스윽 휩쓸린다]. 지나간 지면에 [그을음 자국]이 남는다
④ 종료   — 총구에서 [증기 배출] + 열기 아지랑이
```

- **관통**: 일직선상의 모든 적을 뚫는다. **적이 뭉쳤을 때 한 줄로 증발**시키는 맛
- **강화 시:** **빔 굵기 자체가 커진다** (Lv.1 얇은 선 → Lv.5 **기체보다 굵은 기둥**)
  → 성장이 가장 직관적으로 보이는 무기

#### 7. **레일건**
- **예열 → 초고속 발사 → 즉시 명중**. 빔과 달리 **한 방, 순간**
- 발사 순간 **원형 충격파 링** + 카메라 킥 + **공기가 갈라진 흰 선**이 잠깐 남는다
- **강화 시:** 관통 수 증가 → **Lv.5: 명중 지점에서 2차 폭발**

#### 8. **궤도 폭격 (Orbital Strike)**
- 지상에 **조준 마커**를 찍으면 **1초 뒤 하늘에서 거대한 빛기둥**이 떨어진다
- 화면 전체가 **하얗게 플래시** → 거대 폭발 → 버섯구름
- **강화 시:** 마커 수 증가 → **Lv.5: 융단 폭격 (마커 5개 연속)**

---

### 🔵 유틸 (Control) — 상황을 만든다

#### 9. **그래비티 웰** ★ (핵앤슬래시와 궁합 최고)
- 던진 지점에 **중력장** 발생 → 주변 적들이 **빨려들어 뭉친다**
- **이게 왜 좋은가:** 뭉친 적 무리에 미사일/빔을 꽂으면 **한 방에 화면이 정리된다.**
  → **다른 무기의 화려함을 증폭시키는 무기.** 콤보의 핵심
- **강화 시:** 범위·지속 증가 → **Lv.5: 종료 시 블랙홀 붕괴 폭발**

#### 10. **EMP 필드**
- 광역 슬로우 + 적 사격 봉쇄. **푸른 전기 그물**이 퍼지고 적들에게 **스파크가 튄다**
- **강화 시:** **Lv.5: 감전된 적끼리 전기가 연쇄(체인)**

#### 11. **자율 드론 (Support Drone)**
- 소환하면 **플레이어 주위를 돌며 자동 사격**. 유일한 "자동 공격" 요소
- **강화 시:** 드론 수 증가 → **Lv.5: 드론들이 플레이어와 빔으로 연결되어 지나가는 적을 자른다**

### 3.6 화려함의 규칙 — 지키지 않으면 게임이 부서진다 ★

**이 게임은 수동 조준 슈터다.** 이펙트가 화면을 덮으면 뱀서라이크에선 그냥 시원하지만,
**여기서는 크로스헤어가 안 보여 조준을 못 한다.** 화려함이 게임을 부술 수 있다.

| # | 규칙 | 이유 |
|:---:|---|---|
| **1** | **화면 중앙(크로스헤어 주변)은 비워둔다** | 폭발·플래시는 중앙을 가리지 않게 알파를 낮추거나 **중앙 부분만 투명 처리** |
| **2** | **적의 실루엣은 언제나 보여야 한다** | 이펙트에 적이 묻히면 **피할 수가 없다.** 적에게 **림라이트/아웃라인** 적용 → 이펙트 위로 뜬다 |
| **3** | **적의 공격 예고는 이펙트를 뚫고 보인다** | 적 투사체·경고 표시는 **항상 최상위 렌더**. 화려함보다 우선 |
| **4** | **화면 전체 플래시는 짧게 (0.1초 이내)** | 궤도 폭격 같은 연출도 **눈이 회복될 시간**을 준다 |
| **5** | **모든 VFX는 풀링한다** | `Instantiate` 금지. 미사일 16발 × 트레일 × 폭발이 매 초 터진다 |
| **6** | **강도 슬라이더를 반드시 만든다** | 인스펙터에서 이펙트 강도 0~1 조절. **켜고 끄며 비교**하지 않으면 적정선을 못 찾는다 |
| **7** | **이펙트는 게임플레이에 영향 0** | 폭발 파티클은 판정과 무관. **판정은 코드가, 화려함은 VFX가.** (2.3과 동일 원칙) |

> **성능 상한:** 화면에 수백 마리 + 미사일 수십 발 + 트레일 + 폭발이 동시에 존재한다.
> → **VFX Graph(GPU 처리)** 사용, 파티클 수 상한 설정, **거리 기반 LOD**(먼 폭발은 단순화)
> → 프레임이 떨어지면 **조준이 흔들려 게임이 망가진다.** 성능은 연출의 일부다.

## 4. 성장 — 3택 / 획득 / 강화 / 조합

### 4.1 경험치와 레벨업

- 적 처치 → **경험치 젬** 드롭 → 플레이어가 접촉해 획득 (자석 반경 존재)
- 경험치가 임계치를 넘으면 **레벨업 → 3택 화면**
- **3택 화면이 뜨는 동안 게임 시간은 정지한다.** (싱글 전용이므로 가능)

#### 3택이 뜨는 조건 (2가지)

1. **레벨업** — 상시
2. **특정 조건 달성** — 보스 처치, 특정 시간 도달, 희귀 상자 획득 등
   → 조건 달성 시의 3택은 **더 높은 등급 풀**에서 뽑는다 (보상감)

### 4.2 선택지 카테고리 — 세 가지 병목

| 카테고리 | 푸는 병목 | 정체 | 예시 |
|---|---|---|---|
| **파츠 조립** (Parts) | "적을 못 죽인다" | **화력** | 어깨 미사일 포드, 팔 발칸, 데미지 증가, **무기 슬롯 확장** |
| **장갑 부착** (Armor) | "내가 죽는다" | **생존** | 복합 장갑(최대 HP), 반응 장갑(피격 시 반격), 회피 스러스터 |
| **에너지 자원** (Energy) | **"쏘고 싶은데 쿨이 안 돈다"** | **회전율 · 기동** | **전역 쿨감**, 특정 무기 전용 쿨감, 이동/상승 속도, 착지 쿨감 보너스 강화 |

> **플레이어는 매 레벨업마다 "지금 나를 죽이고 있는 병목이 무엇인가"를 판단해야 한다.**
> 이것이 빌드 설계의 본질이다. 세 카테고리가 서로 **다른 병목**을 풀기 때문에 이 판단이 성립한다.

### 4.3 획득과 강화

- **어떤 아이템을 처음 고르면 = 획득** (해당 파츠/장갑/모듈 장착, Lv.1)
- **같은 아이템을 다시 고르면 = 강화** (Lv.2, Lv.3 … 최대 **Lv.5**)
- 강화 시 수치가 오르며, **특정 레벨에서는 질적 변화**를 준다.
  - 예) 미사일 포드 Lv.3 → 발사 수 2 → 4발, Lv.5 → 유도 성능 획득

### 4.4 조합 (Combination)

- 특정 아이템들이 **각각 요구 레벨에 도달하면**, 3택 풀에 **조합 결과물**이 등장한다.
- 조합 결과는 **재료 아이템을 소비하고** 상위 아이템으로 대체된다.
- 조합품은 강력하지만 **슬롯을 하나로 합치므로**, 언제 조합할지가 또 다른 선택이 된다.

```
예시)
[팔 발칸 Lv.3]      + [고속 냉각기 Lv.2]     →  "가열식 개틀링" (오래 쏠수록 연사↑)
[미사일 포드 Lv.3]  + [조준 보조 헤드 Lv.2]  →  "다중 락온 시스템"
[반응 장갑 Lv.3]    + [출력 증폭기 Lv.3]     →  "오버로드 실드" (피격 시 쿨다운 즉시 회복)
```

> 조합은 **두 카테고리를 가로지르게** 설계한다.
> 그래야 한 카테고리만 파는 걸 막고, 3택에서 **"지금 당장은 필요 없지만 조합을 위해 집는"** 장기 판단이 생긴다.

## 5. 런(Run) 구조 — 20분 시간제

### 5.1 타임라인

```
0:00 ─────────────────────────────────────────────── 20:00
 │        │        │        │        │        │        │
 │      2:00     5:00     8:00    12:00    16:00    20:00
 │        │        │        │        │        │        │
튜토리얼  다수화   중간보스  공중형   포위형   엘리트   최종보스
(소수 잡몹)        (첫 벽)  적 등장   압박    러시     (클리어)
```

- **난이도 축 = 경과 시간.** 시간에 따라 스폰 종류·수·강도가 상승.
- **종료 조건:** 20분 생존(클리어) 또는 사망(실패)
- 두 경우 모두 **통계 화면**으로 이동 (6장)

### 5.2 난이도를 올리는 3가지 손잡이

수만 늘리면 후반이 지루한 청소 작업이 된다. 세 축을 함께 쓴다.

1. **양 (Density)** — 동시 존재 적 수 증가
2. **질 (Quality)** — 더 단단하고 강한 적종 투입
3. **압박 방향 (Pressure)** — **플레이어의 도피처를 좁힌다**
   - 초반: 지상형만 → **공중은 안전지대**
   - 중반: **공중형 적 등장** → 공중도 안전하지 않음
   - 후반: **원거리 사격형 등장** → 착지해서 쿨을 빨리 돌릴 여유도 없음

> **3번이 이 게임 난이도 곡선의 정수다.**
> "안전했던 선택지를 하나씩 빼앗는 것"이 곧 시간 압박이다.
> 특히 후반의 포탑형은 **2.2의 "지상 = 빠른 쿨다운" 보상을 위험하게 만들어**, 저울을 계속 흔든다.

### 5.3 적 아키타입 (v1 최소 세트)

| 적 | 층위 | 행동 | 역할 |
|---|---|---|---|
| **보행 잡병** | 지상 | 플레이어 향해 직진, 근접 접촉 | 물량, 경험치 공급원 |
| **비행 드론** | 공중 | 플레이어 추적, 근접 자폭 | **공중 안전지대 박탈** |
| **포탑형** | 지상(고정) | 원거리 예측 사격 | **착지 방해** (쿨감 보상 견제) |
| **엘리트** | 공중 | 높은 HP, 패턴 공격 | 결정타 무기의 존재 이유 |
| **보스** | 공중 | 다단계 패턴 | 20분 빌드의 최종 시험 |

> **적은 절대 서로 충돌하지 않는다** (`SETUP.md` 5장 — Enemy↔Enemy 물리 OFF).
> 겹침 방지는 **코드 스티어링(분리 벡터)** 으로 처리한다.

## 6. 런 종료 — 통계 화면

클리어/사망 후 표시. **"내 빌드가 무엇을 했는가"** 를 보여주는 게 목적이다.

| 항목 | 내용 |
|---|---|
| 결과 | 클리어 / 생존 시간 |
| 처치 수 | 총 처치, 적 종류별 |
| **무기별 기여도** | 무기별 총 데미지 / DPS / 명중률 — **막대 그래프** |
| **무기 가동률** | 무기별 **"쿨 대기로 못 쏜 시간"** 비율 → 쿨감 투자가 옳았는지 판단 근거 |
| 최고 데미지 | 단일 타격 최댓값 |
| 도달 레벨 | 최종 레벨, 획득 경험치 |
| 최종 빌드 | 장착 파츠·장갑·모듈 목록 (레벨 포함) |
| 지상/공중 체류 비율 | 2.2의 저울을 어떻게 썼는지 |

> **"무기별 기여도"가 이 화면의 핵심이다.** 플레이어가 다음 판에 무엇을 다르게 할지 배우는 유일한 창구다.
> 이를 위해 **모든 데미지 이벤트에 출처(무기 ID)를 실어야 한다.** → 8장 데이터 설계에 반영.

- 싱글플레이 전용

## 8. 시스템 분해 — 코드 설계의 입력

여기서부터는 **구현 계획**이다. `SETUP.md`의 어셈블리 계층에 배치한다.

### 8.1 어셈블리별 배치

```
Core        IDamageable, IPoolable, DamageInfo(+출처 무기 ID), EventBus, ServiceLocator
              GameplayEvents: EnemyKilled, PlayerLeveledUp, DamageDealt,
                              WeaponFired, RunEnded ...

Systems     PoolManager, ComponentPool
              SpawnDirector      — 시간에 따른 적 스폰 (WaveData 소비)
              RunTimer           — 20분 진행, 일시정지 제어
              StatisticsRecorder — 이벤트 구독만으로 통계 수집

Gameplay    [플레이어]
              MechaController    — 입력 → 이동 (비행/보행 전환). ★ 관성·지연 0
              MechaVisuals       — ★ 표현 전담: 상체 포탑 회전, 관절 IK,
                                    스러스터 VFX, 호버 아이들, 기계음
                                    ↳ Controller 상태를 읽기만 한다 (게임플레이에 영향 0)
              MechaAimer         — ICameraRig 에게 시선을 물어 레이캐스트 → 목표점
                                    ※ 발사 판정은 언제나 조준선 기준 (상체 회전과 무관)
              WeaponSlots        — 무기 슬롯 N개 보유, 발사 요청 중계
              Weapon             — 쿨다운 관리 + 투사체 스폰
              CooldownModifier   — 쿨감 집계 (전역/무기별/지상 보너스) ★
              Health             — 체력 (기존)
              UpgradeInventory   — 보유 업그레이드 및 레벨, 조합 판정

            [적]
              EnemyBrain         — 아키타입별 행동 (지상/비행/포탑)
              SteeringBehavior   — 적끼리 겹침 방지 (물리 대신)
              ExperienceGem      — 드롭 → 자석 흡인

            [성장]
              UpgradeService     — 3택 후보 추출, 적용
              RecipeResolver     — 조합 가능 여부 판정

            [카메라] ★
              ICameraRig         — 계약: 시선 원점(Origin) + 시선 방향(Forward)
                                    ↳ MechaAimer 는 이것만 안다 (시점 종류를 모름)
              CameraDirector     — 모드 보유·전환(V 키), 활성 모드에 갱신 위임
               ├ FirstPersonRig  — 콕핏 고정
               └ ThirdPersonRig  — 스프링암 지연 + 충돌 회피
                    ↳ CameraDynamics — FOV/뱅킹/피치/펀치/셰이크 (강도 0~1 조절)

UI          HUD (체력 / **무기별 쿨다운 게이지** / 타이머 / 레벨 / 크로스헤어)
              LevelUpPanel  (3택)
              ResultPanel   (통계)
```

> **HUD에서 가장 중요한 것은 무기별 쿨다운 게이지다.**
> 로테이션이 게임의 핵심인 이상, **"지금 어느 무기가 준비됐는가"** 가 한눈에 보이지 않으면 게임이 성립하지 않는다.

### 8.2 데이터 (ScriptableObject) 스키마 초안

**핵심 설계:** 세 카테고리는 **하나의 공통 베이스**를 상속한다.
그래야 3택 화면·인벤토리·조합이 **카테고리를 몰라도** 동작한다
(개방-폐쇄 원칙 — 새 카테고리를 추가해도 기존 코드를 고치지 않는다).

```
UpgradeData (abstract SO)          ← 3택 화면이 다루는 유일한 타입
 ├─ id, displayName, icon, description
 ├─ category  (Parts | Armor | Energy)
 ├─ rarity    (풀 가중치)
 ├─ maxLevel  (기본 5)
 ├─ levels[]  (레벨별 수치 — 배열이므로 "강화" = 인덱스 증가)
 └─ abstract Apply(MechaContext ctx, int level)   ← 각 카테고리가 구현
      │
      ├─ PartUpgradeData     → 무기 장착 / 슬롯 확장 / 데미지
      ├─ ArmorUpgradeData    → 최대 HP, 피해 감소, 반격
      └─ EnergyUpgradeData   → 쿨감(전역/무기별), 기동력

WeaponData (SO)
 └─ role(Sustain|Burst|Finisher|Control)
    damage, baseCooldown, projectilePrefab, spread, range, homing
    ※ 발사 자원 필드 없음 — 쿨다운이 전부

    [연출 파라미터] ★ 3.4의 화려함은 데이터로 조절한다
    ├─ chargeTime        (빔·레일건의 예고 동작)
    ├─ projectileCount   (미사일 발사 수 — 강화 시 증가)
    ├─ launchStyle       (Direct | VerticalPopup ← 미사일 4단 연출)
    ├─ staggerInterval   (다발 발사·착탄 시차 — "펑" vs "두두두둥")
    ├─ trailVfx / muzzleVfx / impactVfx  (전부 풀링 대상)
    └─ visualScale       (빔 굵기 등 — 강화 시 커진다)

WeaponVfxProfile (SO)   ← 무기 연출을 데이터로 분리
 └─ 강도 슬라이더(0~1), 파티클 상한, 거리 LOD 설정  (3.6 규칙 6·7)

RecipeData (SO)
 └─ ingredients[] { UpgradeData, requiredLevel }  →  result: UpgradeData

WaveData (SO)   ← 이미 존재, 확장
 └─ entries[] { startTime, endTime, EnemyData, spawnInterval, maxAlive, spawnPattern }

EnemyData (SO)  ← 이미 존재, 확장
 └─ archetype(Ground|Flyer|Turret|Elite|Boss), hp, speed, damage, xpValue, prefab
```

### 8.3 실행 흐름

```
Boot ──▶ GameBootstrap: 서비스 등록 (PoolManager, EventBus, ...)
   │
   ▼
Game ──▶ RunTimer 시작 (0:00)
          │
          ├─ SpawnDirector : WaveData 조회 → 시간대별 적 Spawn (풀 경유)
          │
          ├─ [매 프레임] MechaController(비행/보행) / MechaAimer(조준)
          │              WeaponSlots → Weapon.TryFire()
          │                 └ 쿨다운 확인만 (자원 검사 없음)
          │                    └ 통과 시 투사체 Spawn
          │                    └ 다음 쿨다운 =
          │                         baseCooldown
          │                         / (1 + CooldownModifier.Total(weapon))
          │                         / (Grounded ? groundBonus : 1)
          │
          ├─ 투사체 → IDamageable.TakeDamage(DamageInfo{ 출처 무기 ID })
          │              └─▶ EventBus.Publish(DamageDealt)  ──┐
          │                                                    │
          ├─ 적 사망 → EventBus.Publish(EnemyKilled) ─────────┤
          │             └─ ExperienceGem 드롭                  │
          │                                                    ▼
          ├─ 경험치 임계 → PlayerLeveledUp             StatisticsRecorder
          │      └─ Time.timeScale = 0                  (구독만 함 — 전투 코드는
          │      └─ UpgradeService.RollThree()           통계의 존재를 모른다)
          │           ├ 보유 아이템의 강화안 + 미보유 획득안    │
          │           └ RecipeResolver 통과 시 조합안 포함      │
          │      └─ 선택 → UpgradeData.Apply(ctx, level)       │
          │                                                    │
          └─ 20:00 도달 또는 사망 → RunEnded ──────────────────┘
                     └─▶ ResultPanel 이 StatisticsRecorder 조회 → 표시
```

> **설계 요점 ①:** `CooldownModifier`가 쿨감의 **단일 진실 공급원**이다.
> 무기는 "내 쿨다운이 얼마나 줄었는지" 스스로 계산하지 않고 **물어본다.**
> 덕분에 쿨감의 출처(업그레이드/착지/조합품)가 늘어나도 `Weapon` 코드는 바뀌지 않는다.
>
> **설계 요점 ②:** `StatisticsRecorder`는 **이벤트만 구독**한다.
> 나중에 "판 사이 영구 성장"을 붙일 때, **전투 코드를 한 줄도 고치지 않고**
> 같은 이벤트를 구독하는 `MetaProgressService`만 추가하면 된다.

### 8.4 성능 원칙 (SETUP.md 재확인)

- 모든 적·투사체·경험치 젬은 **PoolManager 경유**. `Instantiate`/`Destroy` 금지.
- 적끼리 물리 충돌 없음 → **스티어링으로 분리**.
- 이벤트는 `readonly struct : IEvent` → 힙 할당 최소화.
- 적 AI는 매 프레임 전수 갱신하지 않는다 → **프레임 분산(time-slicing)** 고려.

## 9. 미해결 — 구현 전 결정 필요

| # | 항목 | 선택지 | 비고 |
|---|---|---|---|
| 1 | **착지 쿨감 배율** | ×1.5 / ×2.0 | 너무 높으면 계속 지상에 붙어있게 됨 |
| 2 | 무기 슬롯 초기 개수 | **2개 → 최대 4개** (권장) | 슬롯 확장을 파츠 보상으로 |
| 3 | 3택 리롤 | 있음 / 없음 | 없으면 운의 비중이 커짐 |
| 4 | 맵 형태 | 개활지 / 지형지물 있음 | 포탑형 적의 가치가 달라짐 |
| 5 | 피격 시 무적 시간 | **있음** / 없음 | 물량 게임이므로 있어야 순삭 방지 |
| 6 | 무기 발사 방식 | 클릭당 1발 / 홀드 연사 | 주력(Sustain)만 홀드 허용이 유력 |
| 7 | **기본 시점** | **3인칭** / 1인칭 | 3D 공중전은 주변 파악이 생명 → 3인칭 기본. 1인칭은 옵션 |
| 8 | 카메라 역동성 강도 | 인스펙터 튜닝 | **조준을 방해하면 실패.** 플레이하며 값을 찾는다 |
| 9 | 상체 포탑 회전 속도 | 인스펙터 튜닝 | 느리면 기계답고, 너무 느리면 **총구와 크로스헤어가 따로 노는 게 눈에 띈다** |

## 10. 구현 순서 (권장)

각 단계는 **직전 단계가 플레이 가능해야** 다음으로 넘어간다.

1. **이동 + 카메라** — MechaController(비행/보행 전환)
   → `ICameraRig` + 1인칭/3인칭 전환 → 3인칭 역동 연출 → 크로스헤어 조준
2. **전투 기초** — Weapon + 쿨다운 + 투사체 풀링 + 더미 적(`Health`) + 쿨다운 HUD
3. **적** — EnemyBrain(지상형) + SpawnDirector + 스티어링
4. **성장** — 경험치 젬 → 레벨업 → 3택 → `UpgradeData.Apply` + CooldownModifier
5. **확장** — 무기 4종 / 적 5종 / 조합 / WaveData 20분 타임라인
6. **마무리** — HUD 완성, 통계 화면, 밸런싱

> **1·2단계까지 완성되면 "쏘는 맛"과 "로테이션의 리듬"이 검증된다.**
> 여기가 재미없으면 나머지를 만들어도 소용없다.
> 따라서 1·2단계를 최우선으로 하고, 이 시점에 **반드시 직접 플레이해 확인한다.**
